# 共享目录客户端需求说明：Git 当前快照分支方案

## 1. 背景

需要搭建一套个人或小团队使用的共享目录同步系统。每台 PC 安装一个共享目录客户端，用户在本地普通目录中工作，客户端负责把文件变化同步到中心 Git 服务器。

本方案使用 Git 作为底层数据一致性与冲突判断基础，但不把 Git 仓库直接暴露给用户。服务器侧只维护一个“当前快照分支”，历史提交可以周期性压缩和清理，避免 Git 历史无限增长。

## 2. 目标

1. 多台 PC 可以同步同一个共享目录。
2. 用户只看到普通文件夹，不需要理解 Git。
3. 服务端以 Git 仓库作为权威数据中心。
4. 客户端自动提交、拉取、合并、推送。
5. 遇到无法安全合并的情况时，必须生成冲突副本，不能静默覆盖或丢失数据。
6. 服务端历史可以定期压缩，只保留当前快照和必要的短期恢复信息。
7. 客户端能够识别服务端历史被压缩后的状态，并安全重建本地同步基线。

## 3. 非目标

1. 不提供完整 Git 历史浏览能力。
2. 不支持多人实时协同编辑同一个文件。
3. 不保证二进制文件可以自动合并。
4. 不追求毫秒级实时同步。
5. 不把 Git 命令行暴露给普通用户。
6. 第一阶段不实现复杂权限系统，只考虑单用户或可信小团队。

## 4. 核心原则

1. 服务器是唯一权威状态。
2. 客户端本地目录是工作副本和缓存。
3. 所有本地未同步内容必须先被保护，再执行可能覆盖本地文件的操作。
4. 能证明安全的变化才自动合并，不能证明安全就生成冲突副本。
5. 删除也是一种变更，必须参与冲突判断。
6. Git 历史只是同步工具，不是产品资产，可以被压缩和清理。

## 5. 总体架构

```text
+------------------+        +------------------+
| PC-A 共享目录客户端 | <----> |                  |
+------------------+        |                  |
                            |  Git Server       |
+------------------+        |  current branch   |
| PC-B 共享目录客户端 | <----> |                  |
+------------------+        +------------------+
```

每台 PC 本地结构：

```text
D:\SharedWork\
  用户文件...
  .sync\
    repo\              隐藏 Git 工作仓库或 bare repo
    index.db           客户端同步索引
    staging\           临时写入区
    conflicts\         可选的冲突记录
    logs\              客户端日志
```

服务端结构：

```text
shared-repo.git
  refs/heads/current   当前快照分支
```

## 6. Git 分支模型

服务端只暴露一个主同步分支：

```text
refs/heads/current
```

该分支代表共享目录的最新权威快照。

客户端每次同步时：

1. 把本地变化写入隐藏 Git 仓库。
2. 生成自动提交。
3. fetch 服务端 `current`。
4. 尝试把本地提交与服务端 `current` 合并。
5. 合并成功后 push 到 `current`。
6. push 被拒绝时，重新 fetch 并再次判断。

## 7. 历史压缩模型

服务端可以定期将 `current` 分支压缩成一个新的根提交。

压缩前：

```text
A -- B -- C -- D -- E  current
```

压缩后：

```text
S  current
```

其中 `S` 的文件树内容与压缩前 `E` 完全一致，但没有旧提交历史。

压缩触发条件可以配置：

```text
提交数超过 1000
仓库体积超过 5GB
距离上次压缩超过 7 天
服务端空闲窗口到达
```

压缩完成后，服务端执行 Git 垃圾回收，清理不可达对象。

## 8. 客户端处理历史压缩

客户端需要能够识别服务端历史被重写的情况。

判断方式：

1. fetch 后发现本地记录的 server commit 不再是远端 `current` 的祖先。
2. 远端暴露的 snapshot id 发生变化。
3. push 返回 non-fast-forward，并且普通 rebase/merge 无法建立共同祖先。

处理流程：

```text
1. 暂停同步。
2. 扫描本地工作目录，记录本地未同步变化。
3. 备份本地未同步文件到 .sync/staging。
4. 重置隐藏 Git 仓库到服务端 current。
5. 将服务端 current 渲染到用户工作目录。
6. 把本地未同步变化重新作为新变化应用。
7. 对无法自动应用的文件生成冲突副本。
8. 更新 index.db 的同步基线。
```

客户端不得在未保护本地变化的情况下直接 reset 用户工作目录。

## 9. 客户端同步流程

### 9.1 本地扫描

客户端监听文件变化，同时定期全量扫描兜底。

文件变化来源：

```text
文件新增
文件修改
文件删除
文件重命名
目录新增
目录删除
目录重命名
```

文件 watcher 只能作为触发器，最终判断以扫描和 hash 为准。

### 9.2 本地索引

客户端维护 `.sync/index.db`，记录上一次确认同步成功的文件状态：

```text
path
file_id
content_hash
size
mtime
git_blob_id
last_synced_commit
deleted
last_seen_at
```

`mtime` 只用于快速过滤，不能作为最终冲突判断依据。

### 9.3 三方判断

客户端判断冲突时使用三方状态：

```text
B = base，上次同步成功时的状态
L = local，当前本地状态
R = remote，服务端当前状态
```

基础规则：

```text
L == B 且 R != B：本地没改，应用远端变化
L != B 且 R == B：远端没改，上传本地变化
L == B 且 R == B：无变化
L != B 且 R != B：需要进一步判断是否冲突
```

## 10. 冲突规则

### 10.1 修改冲突

```text
本地修改，远端修改，内容 hash 相同：
  视为无冲突

本地修改，远端修改，内容 hash 不同：
  生成冲突副本
```

冲突副本命名：

```text
report (conflict from PC-A 2026-06-12 15-30).docx
```

### 10.2 删除冲突

```text
本地删除，远端未修改：
  上传删除

本地未修改，远端删除：
  删除本地文件

本地删除，远端修改：
  保留远端文件，记录本地删除冲突

本地修改，远端删除：
  保持远端删除，同时把本地文件保存为冲突副本
```

### 10.3 新建冲突

```text
本地新建 /a.txt，远端也新建 /a.txt，内容相同：
  视为无冲突

本地新建 /a.txt，远端也新建 /a.txt，内容不同：
  路径冲突，生成冲突副本
```

### 10.4 重命名冲突

第一阶段可以把重命名降级为删除旧路径加新增新路径。

后续版本可以通过 `file_id` 或内容 hash 识别重命名：

```text
本地重命名，远端修改：
  可自动合并为新路径加新内容

本地重命名，远端重命名到不同路径：
  生成路径冲突
```

## 11. 删除与回收策略

由于底层使用 Git，删除会体现在 Git 树中。

服务端历史压缩前，删除文件仍可能存在于旧提交中。历史压缩和 GC 后，旧文件对象可以被清理。

建议额外维护短期回收区：

```text
.sync-trash/
  deleted-files metadata
```

默认保留策略：

```text
普通删除文件：保留 7 天
大文件删除：保留 24 小时
冲突副本：不自动删除
```

回收区可以在 Git 外部实现，避免影响当前快照分支。

## 12. 忽略规则

客户端支持 `.syncignore`，语法第一阶段可兼容 `.gitignore` 子集。

默认忽略：

```text
.sync/
.git/
~$*
*.tmp
*.swp
*.lock
Thumbs.db
.DS_Store
desktop.ini
```

被忽略文件不参与同步、冲突判断和删除传播。

## 13. 文件写入稳定性

客户端发现文件变化后，不应立即同步。

建议策略：

```text
文件大小和 mtime 连续 2 秒不变化后再处理
文件被其他进程锁定时延迟处理
大文件上传前计算完整 hash
```

如果文件持续变化，应延迟提交，避免把半写入状态同步到服务器。

## 14. Push 并发处理

多个客户端可能同时 push。

客户端 push 失败时：

```text
1. fetch 最新 current。
2. 重新执行三方判断。
3. 能自动合并则重新提交并 push。
4. 不能自动合并则生成冲突副本。
5. 最多自动重试 3 次。
```

服务端必须禁止非快进 push，普通客户端不得直接 force push。

历史压缩由服务端维护任务执行，维护任务需要有独立权限。

## 15. 压缩任务安全要求

历史压缩必须在服务端维护窗口中执行。

压缩前：

```text
1. 暂停或延迟客户端 push。
2. 记录当前 current commit。
3. 生成同树的新根提交。
4. 原子更新 refs/heads/current。
5. 记录新的 snapshot id。
```

压缩后：

```text
1. 允许客户端重新 fetch。
2. 等待宽限期后执行 aggressive gc。
3. 清理旧对象。
```

宽限期建议：

```text
24 小时
```

## 18. MVP 范围

第一版实现：

1. 单个共享目录。
2. 单个 Git 远端。
3. 自动扫描本地变化。
4. 自动 commit / fetch / merge / push。
5. 修改冲突生成冲突副本。
6. 删除冲突生成冲突副本或冲突记录。
7. `.syncignore`。
8. 服务端 current 分支。
9. 手动触发历史压缩脚本。
10. 客户端识别历史压缩并重建同步基线。

第一版暂不实现：

1. 图形化冲突解决器。
2. 多用户权限。
3. 文件锁。
4. 选择性同步。
5. 远程回收站 UI。
6. 大文件分块上传。

## 19. 验收标准

### 19.1 基础同步

1. PC-A 新建文件，PC-B 能同步得到。
2. PC-A 修改文件，PC-B 能同步得到。
3. PC-A 删除文件，PC-B 能同步删除。
4. PC-A 新建目录和子文件，PC-B 能同步得到完整结构。

### 19.2 冲突处理

1. 两台 PC 同时修改同一文本文件，内容不同，必须生成冲突副本。
2. 一台 PC 删除文件，另一台 PC 离线修改同一文件，恢复同步后不得丢失修改内容。
3. 两台 PC 同时新建同路径不同内容文件，必须生成路径冲突副本。
4. 冲突副本必须继续参与后续同步。

### 19.3 历史压缩

1. 服务端压缩后，`current` 分支文件树与压缩前完全一致。
2. 客户端能识别服务端历史压缩。
3. 客户端在历史压缩后仍能继续同步。
4. 客户端本地未同步文件在历史压缩后不得丢失。
5. 服务端 GC 后仓库体积应下降或不可达对象应被清理。

### 19.4 安全性

1. 客户端异常退出后，下次启动能恢复同步。
2. push 失败不会导致本地文件丢失。
3. merge 失败不会覆盖用户本地修改。
4. `.sync` 目录不会被同步到服务端。

## 20. 推荐开发阶段

### 阶段一：命令行原型

实现最小同步闭环：

```text
scan -> commit -> fetch -> merge -> push -> apply
```

### 阶段二：冲突副本与索引

加入 `.sync/index.db`，完善三方判断和冲突副本生成。

### 阶段三：服务端历史压缩

实现 current 分支快照压缩脚本，并让客户端能自动处理压缩后的重建。

### 阶段四：后台服务化

客户端改为常驻进程，支持文件 watcher、定时扫描、日志和配置。

### 阶段五：体验增强

加入托盘图标、同步状态、冲突列表、手动重试和恢复入口。

## 21. 风险

1. 大型二进制文件频繁修改会导致 Git 对象膨胀。
2. 历史压缩相当于重写分支，客户端必须有专门处理逻辑。
3. Office、IDE、设计软件可能产生大量临时文件，需要持续维护 ignore 规则。
4. Windows 文件锁和大小写行为会影响同步稳定性。
5. 如果用户直接修改 `.sync` 目录，可能破坏客户端状态。

## 22. 总结

本方案把 Git 当成严格的中心化快照存储和冲突判断工具，而不是完整历史系统。服务端 `current` 分支代表共享目录的唯一权威状态，客户端隐藏 Git 细节，为用户提供普通文件夹体验。通过周期性快照压缩，可以控制 Git 历史增长；通过客户端三方判断和冲突副本机制，可以避免静默覆盖和数据丢失。
