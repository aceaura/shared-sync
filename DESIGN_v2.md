# shared-sync v2.0 设计:自适应连接层(打洞优先 / 虚拟组网兜底)

> v1.0.0 已发布(github.com/aceaura/shared-sync,tag v1.0.0)。v2.0 在 `develop` 分支开发。
> 本文档定架构与状态机;具体技术栈见末尾「待定决策」,定了再进入实现。

## 1. 目标(来自需求)

不把同步服务器放公网,而是借一台公网 VPS 作"跳板",打通两个无公网 IP 的私网(如家用 PC ↔ 公司网络),且:

1. **主路径**:VPS 协助 UDP **打洞**,建立两端**直连 P2P**,VPS 只牵线不转发数据。
2. **备用路径**:打洞实在打不通(对称 NAT / CGNAT)时,自动**切换到虚拟组网**(经 VPS/中继的 overlay)。
3. **自动切回**:打洞恢复可用后,自动从备用切回直连。
4. 通用化:不止某一对私网,任意加入的节点都适用。

## 2. 核心设计原则

### 2.1 稳定 overlay IP,路径切换对上层透明(关键)

主备两条路径**共用同一个虚拟内网 IP**(overlay IP,如 `10.77.0.2`)。"切换"只是换底层传输(直连 endpoint ↔ 中继 endpoint),**虚拟 IP 始终不变**。于是:

- shared-sync 引擎的 `server_url` 填的是 overlay IP(如 `http://10.77.0.3:8418/shared.git`),**永远不变,引擎对切换零感知**。
- 切换瞬间最多是一次 TCP 连接重置;而引擎每个同步周期本就是**幂等 + 失败重试**(v1 的核心安全设计),一次抖动只意味着某轮重试,**不丢数据、不需要引擎改任何代码**。

这是 v2 能"加一层而不动 v1 同步内核"的根本。

### 2.2 兜底优先,直连为升级

启动顺序刻意设计为:**先用备用路径(中继)保证立刻可达,再后台尝试打洞;打洞稳定后升级到直连**。好处:连接全程不中断,打洞失败也不影响可用性,只是少了直连的低延迟/高带宽。

## 3. 架构

```text
        家用 PC                         公网 VPS(跳板)                    公司 PC
 ┌────────────────────┐      ┌────────────────────────────┐      ┌────────────────────┐
 │  shared-sync 引擎    │      │  ① rendezvous(信令/发现)    │      │  shared-sync 引擎    │
 │   server_url=overlay │      │     —— 帮两端交换公网映射     │      │                     │
 │        │            │      │  ② relay(中继兜底)          │      │                     │
 │   ┌────▼─────┐       │      │     —— 仅打不通时转发加密流量 │      │   ┌──────────┐      │
 │   │  connd   │◄──────┼──────┼──────────────────────────────┼─────►│  connd   │      │
 │   │ 连接管理 │ 直连优先,中继兜底,自动探测与切回           │      │          │      │
 │   └────┬─────┘       │      └────────────────────────────┘      │   └────┬─────┘      │
 │   overlay 网卡(虚拟IP)│              ▲   直连 P2P(打洞成功时)        │   overlay 网卡    │
 └────────┼────────────┘              └─────────── ✕ ───────────────┘────────┼───────────┘
          └──────────────── 数据直连(打洞 OK)/ 经 relay(打洞失败)─────────┘
```

- **connd(连接管理器)**:每个节点一个,v2 的核心新增组件。维护 overlay 虚拟网卡 + 主备路径,跑状态机,对 VPS rendezvous 注册、做健康探测、执行切换/切回。
- **VPS 两个角色**:`rendezvous`(信令/打洞协调,极小流量)+ `relay`(中继兜底,仅打不通时承载加密流量,**读不到明文**)。
- **shared-sync 引擎/GUI**:基本不动,只新增"连接状态"展示(直连/中继/断开)。

## 4. 状态机(connd 的核心)

```text
            启动
             │  先起备用(中继)保证可达
             ▼
        ┌─────────┐  后台打洞 + 探测                ┌──────────┐
        │ FALLBACK │ ───── 直连稳定达 T_up 秒 ─────► │  DIRECT  │
        │ (中继)   │ ◄──── 直连连续失败 N 次 ──────── │ (打洞直连)│
        └─────────┘                                  └──────────┘
             │                                            │
        每 P 秒重试打洞                              持续心跳保活探测
```

- **FALLBACK → DIRECT(切回/升级)**:每 `P` 秒后台重试打洞;直连建立且健康**连续稳定 ≥ `T_up` 秒(滞后窗口)**才切过去 —— 防止网络抖动导致频繁来回切(flapping)。
- **DIRECT → FALLBACK(降级)**:直连心跳**连续失败 `N` 次**立即切回中继(兜底优先于性能),并在后台继续重试打洞。
- 切换是换底层 endpoint,overlay IP 不变;上层最多感知一次瞬断。

参数(可配):`T_up`(切回滞后,建议 20–30s)、`N`(降级阈值,建议 3–5 次心跳)、`P`(打洞重试周期,建议 30–60s)、心跳间隔(建议 5–10s)。

## 5. 健康探测

- **直连健康**:overlay 内对 peer 的轻量心跳(ICMP/UDP ping 或 WireGuard handshake 时间戳),关注 RTT 与丢包。
- **判定打洞成功**:rendezvous 协调后,两端能在 overlay 上互通心跳即视为直连建立。
- **区分"路径断"与"对端真的下线"**:中继路径也探测;若中继也不通,则是对端离线/VPS 故障,connd 进入重连而非反复切换。

## 6. 与 v1 的集成边界

- 引擎**不改同步算法**;只把 `server_url` 指向 overlay IP。
- GUI 新增「连接」页/状态条:当前路径(直连 P2P / 中继兜底 / 重连中)、对端、RTT、上次切换时间。
- connd 与 sync_cli/GUI 可同机共存(connd 管网络,引擎管同步,互不抢锁)。

## 7. 安全

- overlay 全程加密(WireGuard 噪声协议 / Nebula 证书)。
- relay 兜底时,VPS **只转发加密包,读不到 git 明文**(端到端密钥不在 VPS)。
- rendezvous 仅交换公网映射等元数据,不碰业务数据。
- VPS 仍可叠加防火墙白名单 / 证书签发控制谁能入网。

## 8. 待定决策(进入实现前需确认)——见与用户的确认

1. **技术栈**:自研统一 WireGuard 数据面 + connd / 采用 Nebula(lighthouse 打洞 + relay 兜底,行为天然符合) / 采用 Tailscale·Headscale(自动 direct⇄DERP)。
2. **实现形态**:独立 connd 守护进程,还是由 Dart 引擎管理外部隧道子进程。
3. **fallback 是否与 primary 同一套数据面**(同 overlay IP 换 endpoint,推荐),还是用一套独立虚拟组网做兜底。

## 9. 路线图(待技术栈确认后细化)

- 阶段 0:技术选型 + PoC(两端打洞直连 + 中继兜底,手动切换跑通)。
- 阶段 1:connd 状态机(自动降级/切回 + 滞后防抖)+ 健康探测。
- 阶段 2:VPS 端 rendezvous + relay 的 compose 与一键部署(沿用 v1 的发布/镜像体系)。
- 阶段 3:节点接入与证书/密钥签发脚本(类比 client/daemon 的安装器)。
- 阶段 4:GUI 连接状态页 + 端到端联调(模拟对称 NAT 验证降级、恢复验证切回)。
- 阶段 5:e2e 自动化(用 netns/容器模拟双 NAT + 断网注入)。
